# Principles

- Correctness over speed.
- Produce the smallest complete, production-ready solution.
- Prefer simple, explicit, idiomatic code.
- Optimize for maintainability, clarity, and performance.
- Solve the complete problem, not just the immediate symptom.

# Coding session initialization

At the start of a newly created coding session:

1. Attempt to rename the session using:

   `[<project>] <description>`

2. Attempt to pin the session.

3. Before making any edits, inspect the Git working tree and current delivery state.

4. For implementation work:
   - If the current request explicitly continues work already covered by an existing GitHub Issue, use its matching branch and existing pull request when one exists.
   - Otherwise, when preflight proves that the current branch is the clean default branch, matches the refreshed remote default branch, and no issue, branch, or pull request already covers the requested work:
     - Do not ask whether to create a new issue, branch, and pull request or continue existing work.
     - Create a new GitHub Issue.
     - Create a branch named `GH-<issue-number>` from the refreshed remote default branch.
     - Perform all implementation work on that branch.
   - Otherwise, when work is already in progress and the requested implementation is not covered by its issue or pull request, ask whether to create a new issue, branch, and pull request or continue on the existing branch and pull request.
     - If the user chooses a new lane, create a new GitHub Issue and exact `GH-<issue-number>` branch.
     - If the user chooses to continue, create or reuse a separate GitHub Issue for the additional work, keep the existing branch and pull request, scope the new commits to that issue, and add its reference to the pull request.

Rules:

- Use the current repository name as `<project>`.
- Keep `<description>` lowercase and at most four words.
- Name a new implementation branch exactly `GH-<issue-number>`.
- The GitHub Issue exists only to track the unit of work and provide a stable identifier for the branch, commits, and pull request. Do not treat it as the source of implementation requirements.
- Continuing additional work on an existing pull request changes issue-to-commit traceability only. It does not create or switch to a replacement branch or pull request.
- Never stop or delay work if renaming or pinning is unsupported or fails.

# Execution environment

- Work in the existing checkout when it already owns the requested lane.
- For a separate lane, reuse or create `~/worktrees/<owner>/<repository>/<lane>`; never put a worktree inside a repository.
- Use exact `GH-<issue-number>` for durable issue work.
- Use uppercase detached `PR-<number>` only for temporary pull-request inspection. Reuse the pull request's same-repository head branch for writable repair.
- Before creating a worktree, inspect registered worktrees and reuse the exact branch worktree when one exists.
- Use native `git worktree` commands and ordinary filesystem operations as the portable authority for lane creation, reuse, detached inspection, repair, exact-path validation, movement, pruning, and removal; do not require `git-wt`, an alias, or another wrapper.
- Optional wrappers are manual conveniences only and must preserve the same path and safety contract; load `docs/references/worktrees.md` for the native workflow and optional command usage.
- Keep the root checkout on the protected default branch and work directly in the assigned durable lane.
- Never stash, reset, clean, force-remove, or delete a branch to create or clear a worktree.
- Link the invoking checkout's `.env` into writable lanes by default when it exists, using only an exact verified symlink; omit the link when isolation is required.
- Never copy `.env` contents or automatically share `.envrc`; worktree tooling does not manage runtime services, databases, ports, Temporal state, processes, or sibling repositories.
- Treat working files, the index, and `HEAD` as worktree-local; treat objects, refs, remotes, configuration, and stash entries as shared clone state.
- Before making edits, inspect the selected worktree and preserve changes that were not created for the current request.
- Keep the GitHub Issue, branch, commits, and pull request aligned in scope, except for explicitly continued additional work whose new issue-scoped commits remain on the existing branch and pull request.

# Execution

- Read project instructions and local conventions before editing.
- Investigate the relevant codebase before making changes.
- Prefer evidence from the repository over assumptions. Infer intent from existing code, tests, documentation, and project conventions whenever possible.
- Own the requested outcome, not just the requested edit. Complete all code, tests, configuration, documentation, refactoring, and supporting changes necessary for the result to work correctly.
- Resolve routine implementation decisions independently using repository evidence, project conventions, and the simplest reasonable approach.
- Continue pursuing the requested outcome until it is complete or blocked by missing information, permissions, or external dependencies.
- If an approach fails, identify the cause, revise the approach, and continue.
- If blocked, complete all unblocked work first, then clearly report:
  - the blocker,
  - the supporting evidence,
  - why it cannot be resolved locally, and
  - the smallest action required to proceed.
- Ask questions only when the answer would materially change the implementation and cannot be determined from the repository, project conventions, available tooling, or reasonable inference. Otherwise, choose the best-supported approach and continue.

# Debugging

- Identify the root cause before implementing a fix.
- Prefer fixes that eliminate the underlying issue rather than masking it.
- Remove obsolete or unnecessary code when appropriate.

# Design

- Default to the simplest solution that completely satisfies the requirements.
- Preserve existing architecture, behavior, interfaces, and coding conventions unless changing them is necessary to satisfy the current request.
- When choosing between multiple reasonable approaches, select the simplest solution that satisfies the current request. Briefly explain the tradeoffs only when they materially affect the implementation or future maintenance.

# Validation

- Review all changes before finishing.
- Run all relevant tests, builds, type checks, linting, formatting, or other project validation appropriate for the affected code.
- Validate the requested outcome end-to-end whenever practical.
- Do not treat a successful build or a narrow passing test as sufficient evidence that the request is complete.
- Resolve issues introduced by your changes before considering the request complete.
- If validation cannot be performed, clearly state what could not be validated and why.

# Repository memory completion

- After implementation and validation, load `docs/references/rules/constitution-curation.md` when present and curate durable rationale into the appropriate repository documents.
- Every implementation final response must include `Repository Memory`, `Decision`, `Rationale`, and `Artifacts` in addition to outcome, validation, and remaining risks.

# Pull request

- After completing implementation and validation, open a pull request for review when none exists; otherwise update the existing pull request.

# Communication

- Keep progress updates brief and limited to significant findings, completed milestones, blockers, or changes in approach.
- Lead final responses with:
  1. Outcome
  2. Validation performed
  3. Remaining risks, assumptions, or recommended follow-up work
